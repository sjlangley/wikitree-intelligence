# Test Plan

Generated by /plan-eng-review on 2026-04-06

## Affected Pages/Routes

- `/login` — verify Google sign-in creates an app session and restores it on return
- `/settings/integrations/wikitree` — verify WikiTree auth succeeds, expires cleanly, and only exposes permitted private data
- `/imports/new` — verify GEDCOM upload, validation, and import job creation
- `/imports/:id` — verify staged progress, pause, resume, failure, and recovery states
- `/matches/:importId` — verify anchor selection, candidate review states, evidence packets, and traversal continuation
- `/sync-review/:importId` — verify matched profiles with extra GEDCOM facts appear for later review

## Key Interactions to Verify

- Sign in with Google, close the app, reopen it, and confirm the same import/review state is restored
- Connect a WikiTree account, fetch private-but-permitted profile data, then handle expired auth without silent failure
- Start a large GEDCOM import, pause it, resume it, and confirm no duplicate person/relationship rows are created
- Anchor one GEDCOM person to one WikiTree profile and confirm outward traversal continues through confident matches
- Accept, reject, and defer candidate matches and confirm the decision history persists
- Confirm a matched profile with extra GEDCOM facts lands in the later sync-review queue instead of being updated during import

## Edge Cases

- Malformed GEDCOM file
- GEDCOM with alternate spellings, missing dates, duplicate names, and incomplete relatives
- WikiTree session expiry mid-import
- Import stage failure after partial persistence
- Previously rejected candidate resurfacing after re-import
- Confident surrounding relatives imply a downstream auto-accept, but one material field conflicts
- Private-data-derived evidence packet is shown only after authenticated WikiTree access

## Critical Paths

- Google sign-in -> backend session -> GEDCOM upload -> import created
- Import stage pipeline -> checkpoint written -> app closes -> resume from checkpoint
- Anchor person -> fetch candidates -> classify likely/maybe/no-safe-match -> persist snapshot review record
- Confident existing match -> continue traversal outward -> surface missing match or conflicting data
- Matched person with extra GEDCOM facts -> enter later sync-review queue -> reviewer sees additive vs conflicting facts
