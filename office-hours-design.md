# Design: WikiTree Match Workbench

Generated by /office-hours on 2026-04-06
Branch: main
Repo: sjlangley/wikitree-intelligence

## Problem Statement

Build a personal genealogy workbench that starts with a GEDCOM, then helps reconcile
people into WikiTree with better matching and better sourcing than the current manual
GEDCOM import workflow. The core job is not bulk import. The core job is finding likely
existing people in WikiTree, including alternate spellings and incomplete links, before
creating duplicate profiles. Once a match is strong enough to trust, the system should
keep moving outward through the tree to find the real work: missing matches and data
differences that still need attention.

## What Makes This Cool

The magic moment is discovery: connecting a profile that already exists in a public tree
to your own line when you did not know it was there. The product should feel less like
"uploading genealogy data" and more like a genealogy detective that surfaces hidden
branches safely. The second magic moment is seeing the tool skip over obvious resolved
matches and focus your attention on the places where the trees diverge.

## Constraints

- This is a personal tool first, optimized for one power user rather than a generic
  consumer workflow.
- The MVP should start GEDCOM-first, not Google Drive-first.
- GEDCOM imports may be large, so import and reconciliation work must be resumable across
  sessions.
- WikiTree is the collaboration target.
- The user should be able to authenticate with WikiTree so the app can access private
  data the user is allowed to see.
- The app should use Google authentication for its own user login so progress, match
  memory, and authenticated sessions are preserved between runs.
- Authority should come from cited records, not Wikipedia and not unsourced tree data.
- WikiTree has a one-profile-per-person philosophy, so duplicate creation is a critical
  failure mode.
- The tool should stay human-in-the-loop before any profile creation or edit suggestion
  is applied.
- The system should preserve confirmed matches across sessions so work compounds instead
  of restarting from scratch.

## Premises

1. The real job is duplicate-safe discovery into WikiTree, not bulk import.
2. The MVP should start GEDCOM-first, not Google Drive-first, because scans, OCR, and
   photo extraction are a second hard problem.
3. WikiTree is the collaboration target, but authority should come from cited records,
   not Wikipedia and not unsourced tree data.
4. Version 1 should be a local-first, human-in-the-loop review tool that suggests
   matches and citation drafts before any profile creation.
5. Version 1 should maintain a durable registry of confirmed GEDCOM-to-WikiTree matches
   and use those confirmed anchors to simplify future matching decisions.
6. When the tool is confident that a GEDCOM person already matches a WikiTree profile, it
   should continue outward automatically and focus review attention on missing matches
   and conflicting data.

## Approaches Considered

### Approach A: GEDCOM Match Workbench

A local app that imports one GEDCOM, lets the user anchor one known person to one
WikiTree profile, then walks outward and scores likely WikiTree matches for relatives.

Why it works:

- Fastest path to the core payoff
- Keeps review in the loop
- Directly attacks the current manual matching pain

Tradeoffs:

- Narrower than the long-term archive dream
- Citation drafting will start simple
- Google Drive, OCR, and scan ingestion wait for later phases

### Approach B: Identity Graph Engine

Build a richer local graph of people, relationships, sources, alternate spellings, and
candidate matches, then layer review workflows on top.

Why it works:

- Better long-term architecture
- Cleaner support for source provenance and evidence packets
- Easier to add archive ingestion later

Tradeoffs:

- Slower to first useful result
- Easier to overbuild
- More design complexity before the first real user win

### Approach C: Gramps Sidecar

Use Gramps as the genealogy data backbone, then build a sidecar tool or plugin for
WikiTree matching, duplicate-risk review, and source packet generation.

Why it works:

- Reuses a mature genealogy model
- Avoids rebuilding GEDCOM and citation fundamentals
- Potentially faster path to a capable prototype

Tradeoffs:

- Inherits Gramps constraints and UX
- Harder to shape into a clean custom review experience
- Less attractive if the end state is a custom React app

## Recommended Approach

Choose Approach A.

It gets to the real payoff fast: import a GEDCOM, anchor one known person to WikiTree,
walk outward, and show likely matches with enough evidence to avoid creating the wrong
profile. This preserves the fun part of the project, discovering unknown branches,
without burying the work under archive ingestion and graph-engine ambition.

Approach A should still borrow one idea from Approach B: every candidate should carry an
"evidence packet" that explains why the match is plausible. That packet becomes the unit
of trust in the system.

## Version 1 Boundaries

In scope for version 1:

- Local app login or user profile identity
- Local GEDCOM import
- WikiTree authentication for the local user
- Resumable GEDCOM import jobs with preserved progress
- One anchor person mapped to one known WikiTree profile
- Relative and ancestor expansion from that anchor
- Persistent storage of confirmed matches and rejected candidates
- Candidate WikiTree matching with transparent confidence states
- Evidence packet generation for each candidate
- Read-only review workflow plus exportable handoff artifact
- Automatic acceptance of obvious downstream matches when they are implied by already
  confirmed surrounding matches and there are no material conflicts
- Outward traversal that keeps progressing past confident existing matches so the review
  queue concentrates on gaps and mismatches
- A separate post-import review queue for profiles where GEDCOM contains useful data that
  is missing from WikiTree

Out of scope for version 1:

- Automatic WikiTree profile creation
- Automatic WikiTree edits
- Google Drive ingestion
- OCR pipelines for scans and photos
- Full archive organization
- Multi-source record crawling beyond a small number of adapters
- Heavy AI dependence as a requirement for core matching correctness

## Open Questions

- Which WikiTree authentication flow, APIs, and rate limits matter for search, private
  profile lookups, and possible write flows?
- Should the first version stop at review, or export a reviewable CSV/JSON artifact for
  later WikiTree entry?
- Which external public sources are most worth consulting after WikiTree itself:
  FamilySearch, public archives, or something else?
- How much citation drafting is possible from GEDCOM data alone, versus requiring record
  lookups from other services?
- What is the minimum evidence threshold for "likely match" versus "maybe"?
- What exact rules qualify a match as "obvious enough to auto-accept" based on already
  confirmed relatives and non-conflicting fields?
- Which GEDCOM facts are worth surfacing in a later sync-review queue:
  names, dates, places, relationships, notes, or sources?

## Success Criteria

- Allow the user to sign into the app and return later without losing import progress,
  match memory, or authenticated WikiTree context.
- Import one GEDCOM locally.
- Allow the user to authenticate with WikiTree and read the private profiles and
  relationships they are permitted to access.
- Pause and resume a large GEDCOM import without restarting from the beginning.
- Allow the user to map one GEDCOM person to one known WikiTree profile.
- Persist confirmed and rejected matches so future runs do not repeat the same work.
- Traverse outward to relatives and ancestors from that anchor.
- For each candidate, show one of three review states:
  likely match, maybe match, or no safe match.
- For each candidate, show a transparent score and reasons.
- When an existing GEDCOM-to-WikiTree match is already strong enough, mark it resolved
  and continue outward automatically without blocking the traversal.
- Produce at least one real match the user would probably have missed manually.
- Surface at least one meaningful gap:
  either a missing person match or a material data discrepancy between GEDCOM and
  WikiTree.
- Build a reviewable list of already-matched profiles where GEDCOM has additional facts
  or sources that do not exist in WikiTree yet.
- Automatically accept at least some clearly implied matches when prior confirmed
  surrounding matches make the identity unambiguous and no conflicts are present.
- Avoid creating or recommending a duplicate with weak evidence.
- Generate a draft citation/evidence packet for candidates where the source data is
  strong enough.

## Distribution Plan

Version 1 should be a local-first desktop-style web app:

- Backend: Python service for GEDCOM parsing, matching, and evidence assembly
- Frontend: React + TypeScript review interface for anchors, candidates, and evidence
  packets
- Storage: a real local database, not just flat files, so match memory and evidence can
  persist cleanly
- Delivery: run locally via Docker Compose during development, or a dev server if that
  proves simpler early on

For early development, no public deployment is required. The tool can run entirely on a
local machine against local GEDCOM files. If the project becomes something worth reusing,
the next distribution step is a GitHub repository with reproducible local setup and a
small sample GEDCOM for testing.

The local storage choice should still point toward a future hosted deployment. A good
default is PostgreSQL in Docker Compose for version 1, because it handles relational
people, relationships, evidence packets, and match state cleanly, and it can later move
to a cloud platform with minimal conceptual churn. SQLite is acceptable only if speed of
initial prototyping clearly outweighs the migration cost.

Suggested version 1 stack shape:

- monorepo layout with runnable apps under `apps/`
- `frontend`: React + TypeScript
- `api`: Python service
- `db`: PostgreSQL
- `auth`: Google sign-in for app identity
- `compose`: local orchestration for repeatable setup and future cloud portability

Recommended repo shape:

```text
apps/
  api/
    tests/
  web/
    src/
    tests/
e2e/
migrations/
```

The app should also have a minimal user model, even in local-first mode, so it can
associate:

- resumable import jobs
- confirmed, rejected, deferred, and derived matches
- WikiTree authenticated session state
- preferences and future source connections

Authentication should be handled as a first-class backend concern:

- the local app should support signing in to WikiTree with the user account
- session or token material should be stored securely
- private WikiTree data should only be fetched for the authenticated user and only used
  within the local app context
- the design should assume a later hosted version will need stricter secret management
  and clearer trust boundaries

For app identity, use Google auth as the default version 1 login because it is easy to
implement in a SPA and gives a clean path to a later hosted deployment. This does mean
the app depends on an external identity provider even when the rest of the workflow is
local-first.

## Evidence Packet

Each candidate review should produce a compact evidence packet containing:

- Candidate GEDCOM person summary
- Candidate WikiTree profile summary
- Matched fields:
  names, alternate spellings, dates, places, relatives
- Conflicting fields:
  anything that lowers confidence
- Source provenance:
  where each fact came from in the GEDCOM or downstream record source
- Reviewer recommendation:
  likely, maybe, or no safe match

Version 1 citation drafting should be conservative. If the GEDCOM contains enough source
detail, draft a structured citation note. If it does not, fall back to provenance notes
and a prompt for manual record lookup rather than pretending the source is stronger than
it is.

## Match Memory

Version 1 should maintain local state for:

- Confirmed GEDCOM person to WikiTree profile matches
- Rejected candidate matches
- Deferred or uncertain matches that need another look
- The reasons and evidence used for each decision
- Import job progress and reconciliation checkpoints

This state is not just convenience. It is part of the product. The tool should get
smarter as you use it by remembering what has already been settled.

Import progress is part of that memory too. A large GEDCOM should be processed as a job
with checkpoints so the user can stop, come back later, and continue instead of paying
the full cost again.

Confirmed matches should also act as traversal bridges. Once the system trusts that a
GEDCOM person and WikiTree profile are the same person, it should use that resolved node
to continue exploring outward rather than forcing unnecessary review on already-settled
identity work.

The first automation rule should be conservative:

- If a person becomes an obvious match because adjacent relatives are already confirmed
  on both sides, and the names, dates, and places do not materially conflict, the tool
  may auto-accept that match and mark it as derived from prior confirmed links.
- If any meaningful conflict exists, fall back to manual review.

This keeps the system useful without pretending uncertain genealogical inference is
fact.

The persistence model should be backed by the main application database so that:

- confirmed matches can be reused across sessions
- derived matches can be traced back to the prior confirmations that implied them
- rejected candidates are not resurfaced endlessly
- evidence packets remain attached to the decisions they justified

Private-data-derived facts should also record that they came from authenticated WikiTree
access, so it is clear which evidence depended on the signed-in user's visibility.

## AI Role

AI should be optional in version 1, not foundational.

Version 1 can succeed with deterministic parsing, fuzzy matching, and rule-based
confidence scoring. If AI is used early, it should help with narrow tasks like:

- Explaining why a candidate was suggested in plain English
- Summarizing messy provenance into a cleaner evidence note
- Suggesting alternate spellings or place-name normalizations for review

The core identity decision should not require AI to be trustworthy in version 1.

## Review Focus

The review queue should prioritize what is actually interesting:

- missing matches where a GEDCOM person likely exists in WikiTree but is not yet linked
- conflicting facts where GEDCOM and WikiTree disagree on names, dates, places, or
  relationships
- low-confidence cases where human review is required
- matched profiles where GEDCOM has useful facts or sources that WikiTree does not yet
  contain, for later sync review

It should de-prioritize already-resolved confident matches except as supporting context
for continued outward traversal.

## Later Sync Review

Import and identity matching should stay separate from profile enrichment.

If a person already matches between GEDCOM and WikiTree, but the GEDCOM contains useful
facts, sources, or relationship details that WikiTree does not yet have, the app should
add that profile to a later sync-review list instead of trying to update it during
import.

This later sync-review list should show:

- the matched GEDCOM person and WikiTree profile
- which facts exist only in GEDCOM
- which sources or provenance support those facts
- whether the difference looks additive or conflicting
- a human review action for deciding whether to carry the data into WikiTree later

This keeps import focused on identity resolution while still creating a path for useful
data improvement work after the tree is mapped.

## Authentication Boundary

Version 1 should treat WikiTree authentication as read access for better matching, not as
permission to write automatically.

The safe boundary for version 1 is:

- sign in to WikiTree
- read public and user-visible private data
- use that data to improve candidate matching and evidence packets
- keep final acceptance and any eventual WikiTree edits under explicit user control

This keeps authenticated access useful without turning version 1 into an unsafe sync
engine.

Separately, the app should have its own user login boundary via Google auth. That login
must be enough to bind together:

- resumable import jobs
- persisted match memory
- stored evidence packets
- WikiTree authenticated access

Without this, large-import recovery and private-data workflows become brittle.

## Next Steps

1. Define the candidate matching model:
   given name, surname, alternate spelling, birth year range, death year range, place,
   relatives, and confidence weighting.
2. Define the anchor workflow:
   import GEDCOM, select one person, match to one WikiTree profile, expand outward.
3. Define the local match-memory model:
   confirmed matches, rejected matches, deferred matches, and derived auto-accepted
   matches.
4. Define the evidence packet shape:
   why this matched, what data agreed, what data conflicted, and what source text is
   available.
5. Define the conservative auto-accept rules:
   when prior confirmed relatives make a new match obvious enough to accept.
6. Decide whether version 1 only reviews matches or also prepares a structured handoff
   artifact for WikiTree entry.
7. Decide whether AI appears in version 1 only as an explainer layer, or waits until
   version 2.
8. Pick the initial source adapters after WikiTree:
   likely GEDCOM first, then FamilySearch or other public record sources.
9. Decide the persistence stack:
   PostgreSQL in Docker Compose by default, or a lighter temporary option if startup
   speed matters more than migration cost.
10. Confirm the WikiTree authentication approach:
   how local sign-in works, how tokens or sessions are stored, and which private fields
   are available once authenticated.
11. Define the app user model:
   how Google auth works and how imports, match memory, and WikiTree auth are attached
   to that user.
12. Define the resumable import job model:
   checkpoints, restart behavior, and what counts as completed versus deferred work.
13. Define the outward traversal rules:
   when a confident match is considered resolved enough to keep walking outward, and how
   missing matches and conflicting facts are promoted into the review queue.
14. Define the later sync-review model:
   how matched profiles with extra GEDCOM data are queued, compared, and prepared for
   manual WikiTree update decisions.
